# v0.2.0 Release Review

Review date: 2026-05-17

Scope: release readiness for the Rust CLI v0.2.0 and the first PyPI release of the Python helper package. The R package was reviewed only as an in-repo helper package, not as a CRAN release candidate.

## Verification Performed

- Re-read the repo instructions and review guidance in `.AI/collaboration.md`, `.AI/writing_style.md`, `.AI/testing.md`, `.AI/api_boundaries.md`, and `.AI/scientific_code.md`.
- Reviewed Rust package metadata, CLI feature boundaries, generated-output writers, Zarr schema writers, settings sidecars, release docs, website docs, Python package metadata/loaders, R package metadata/loaders, and downstream compatibility workflow structure.
- Ran `cargo check --all-features --examples`. It passed.
- Ran `cargo package --allow-dirty --list` to inspect the crates.io package boundary. The Rust package boundary looks intentional: Rust source, tests, README, CHANGELOG, LICENSE, lockfile, and logo are included; website, Python package, R package, examples, and internal docs are excluded.
- Did not run tests, per project instruction. The downstream Zarr tests were already running in the PR and should remain a release gate.

## Summary Judgment

I did not find a core Rust/Python/R schema mismatch for the v0.2.0 public Zarr outputs. The Rust writers for `midpoint_profiles` and `end_motif_counts` and the Python/R loaders are aligned at the structural level: schema names and versions, axis arrays, row metadata, dense/sparse end-motif modes, and basic shape validation match.

After follow-up review, the stale checked-in generated website files are not a release finding for the deployed docs path. The docs workflow runs `./website/scripts/generate_cli_docs.sh` before building, and `website/package.json` also has `prestart` and `prebuild` hooks that regenerate the generated files. This only matters if someone publishes from checked-in generated files without running the normal build path.

- Fixed stale scaling-factor filenames in the hand-written website guides. The examples now use `.fragment_counts.scaling_factors.tsv` for `fragment-count-weights` outputs and `.coverage.scaling_factors.tsv` for `coverage-weights` outputs.

## Need To Fix Before Release

### 1. Update Python package install docs before PyPI upload

Evidence:

- `py-cfdnalab/pyproject.toml:1` through `py-cfdnalab/pyproject.toml:3` define the PyPI project as `cfdnalab` version `0.1.0`.
- `py-cfdnalab/README.md:13` through `py-cfdnalab/README.md:23` only document installation from GitHub.

Why this blocks the PyPI release:

The PyPI project page will render the README. If it only tells users to install from GitHub, the first PyPI release will publish with stale installation instructions.

Fix:

Put the PyPI install path first once the PyPI project exists:

```bash
pip install cfdnalab
```

Then keep the GitHub install command as the development install path. Publishing Python `cfdnalab` as `0.1.0` is consistent with the internal versioning policy, because helper package versions do not need to match the Rust CLI.

### 2. Keep downstream compatibility and package-build checks as hard gates

Evidence:

- The local Rust check passed, but no tests were run per instruction.
- The downstream Zarr tests were pending in the PR during this review.
- I did not build the Python sdist/wheel in this review.

Why this blocks release:

The biggest v0.2.0 promise is the new output/readers contract. The release should not go out until that contract is validated from built artifacts, not just source inspection.

Fix:

Before publishing:

```bash
cargo check --all-features --examples
cargo package --list
python -m build py-cfdnalab
twine check py-cfdnalab/dist/*
```

Then install the built Python wheel in a clean environment and run the downstream compatibility checks. The PR downstream tests should be green before tagging.

## Not Release Blockers From This Review

- Rust crate metadata is set to `0.2.0` in `Cargo.toml:1` through `Cargo.toml:5`, and the package include boundary looks appropriate for the Rust crate.
- `CITATION.cff` was checked earlier in the review and is aligned with v0.2.0.
- Python helper version `0.1.0` is not itself a blocker. `.AI/docs/future/release_checklist.md:30` through `.AI/docs/future/release_checklist.md:49` explicitly says Rust, Python, and R versions do not need to match.
- The generated website docs are regenerated by the docs workflow/build hooks, so stale checked-in generated files are not a deployed-site blocker under the normal release path.
- The helper-package wording concern is specifically about GC package Zarr stores. If the intended user contract is that users do not read GC packages directly, this is not a release blocker. It is still worth keeping public wording precise when touching the docs.
- `cfdna lengths` uses `ApplyGCArgFileOnly`, so its `length_settings.json` `gc_correction_used` flag based on `gc_file.is_some()` matches the command's actual GC-correction mode. I did not find a `--gc-tag` sidecar bug there.
- The current `ends` settings sidecar is intentionally narrow and records motif-definition settings, not full provenance. That can be expanded later, but I would not block v0.2.0 on it unless you want settings JSON to become a reproducibility manifest.

## Future Work

- Publish a public schema compatibility page based on `.AI/docs/future/schema_compatibility.md` once v0.2.0 is out, so users can answer "which helper reads this output?" without reading internal docs.
- Add PyPI package polish after the first release: classifiers, `Project-URL` entries for docs and changelog, a visible `__version__`, and a documented release command. Consider bounding `zarr` as `zarr>=3,<4` if you want to avoid a future major-version break landing silently.
- Harden Python loader validation for hostile or hand-edited stores: blacklisted-fraction ranges, half-open row intervals, duplicate/sorted sparse COO coordinates, finite/nonnegative counts, and motif-label character validation.
- Harden R loader validation for large genomic coordinates if the zarr reader returns numeric doubles where integer precision could matter.
- Decide whether the Rust library API should remain broad. `src/lib.rs:3` through `src/lib.rs:5` currently expose `commands`, `error`, and `shared`; `.AI/api_boundaries.md` prefers a smaller curated public API. This is not a CLI release blocker, but it matters once downstream Rust users depend on internals.
- Before enabling or documenting more non-default commands, do a focused panic/error-path audit for optional and experimental command features. The default release-facing commands mostly use `anyhow` errors, but optional command surfaces should get the same scrutiny before they become user promises.
- Consider making multi-artifact finalization all-or-nothing at the command level. Individual files are written through final-output helpers, but a later artifact rename failure can still leave earlier final artifacts visible. This is acceptable for v0.2.0, but stricter transaction semantics would be cleaner for workflow engines.
- Expand settings sidecars only if there is a concrete downstream need. If provenance becomes a release promise, add correction/scaling/blacklist inputs consistently across commands instead of growing command-specific ad hoc JSON.

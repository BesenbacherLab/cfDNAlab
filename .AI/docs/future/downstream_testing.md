# Downstream Compatibility Testing

This note sketches CI-only tests for public output formats. These tests should
verify that files written by `cfDNAlab` can be loaded by the downstream tools
users are likely to use, without making those tools part of the normal local
Rust development environment.

## Why This Belongs In Separate CI

The normal development loop should stay Rust-focused:

```text
cargo check
cargo check --tests
```

R packages, Python scientific stacks, system libraries, and Bioconductor release
timing are a different dependency surface. They are still important because the
public output contract is only useful if real downstream readers can load it.
The right compromise is a separate GitHub Actions workflow that runs on demand,
on pushes to `main`, and on PRs that touch public output code or downstream
fixtures.

## Implemented Layout

```text
downstream_tests/
  README.md
  requirements-python.txt
  python/
    conftest.py
    test_end_motif_zarr_cfdnalab_package.py
    test_end_motif_zarr_python.py
    test_midpoint_zarr_python.py
    test_midpoint_zarr_cfdnalab_package.py
    test_midpoint_zarr_xarray.py
    test_midpoint_zarr_dask.py
  R/
    common.R
    test_cfdnalab_r_package.R
    test_end_motif_zarr_cran_zarr.R
    test_end_motif_zarr_rarr.R
    test_midpoint_zarr_cran_zarr.R
    test_midpoint_zarr_rarr.R
tests/
  generate_downstream_zarr_fixtures.rs
```

`generate_downstream_zarr_fixtures.rs` creates small BAM/interval inputs, runs
the real `cfdna midpoints` and `cfdna ends` commands, and leaves the resulting
Zarr stores under `downstream_tests/tmp`. This is required: the downstream suite
exists to catch cfDNAlab output-schema drift, so the files consumed by Python
and R must be written by cfDNAlab itself.

The Python and R tests should fail when the cfDNAlab-generated fixture is
missing. They should not create a hand-authored Zarr store as a fallback.

The same ignored fixture test also runs `cfdna ends` to create dense global,
sparse windowed, and sparse grouped Zarr stores. Ends compatibility has a
different success condition: reconstructing motif matrices and sparse matrices
with row and motif metadata intact.

## GitHub Actions Workflow

Use a separate workflow:

```text
.github/workflows/downstream-zarr.yml
```

Suggested triggers:

```yaml
on:
  workflow_dispatch:
  push:
    branches:
      - main
  pull_request:
    paths:
      - "src/commands/midpoints/**"
      - "src/commands/ends/**"
      - "src/shared/**zarr**"
      - "tests/generate_downstream_zarr_fixtures.rs"
      - "tests/fixtures/**"
      - "downstream_tests/**"
      - "py-cfdnalab/**"
      - ".github/workflows/downstream-zarr.yml"
```

Run the workflow on pushes to `main` and relevant pull requests. Do not schedule
it with cron. The suite should be small enough that running it on code changes is
the right signal, and scheduled package-drift failures are likely to be noise
unless the output code is being changed.

Keep the workflow optional at first. Once the midpoint Zarr writer and supported
downstream package set are stable, it can become required for PRs that touch the
Zarr writer or output schema.

The workflow jobs are:

- Fixture generation: Rust ignored integration test that runs `cfdna midpoints`
  and `cfdna ends`
- Python: `zarr`, `xarray`, and `dask.array`
- R: CRAN `zarr` and Bioconductor `Rarr`

The R job installs `libblosc-dev` and `libzstd-dev` before installing R
packages. Those libraries make compression support explicit instead of relying
on the GitHub runner image.

Do not let package-sprawl leak into the user docs. CI can test several readers,
but the guide should show one recommended Python path and one recommended R
path.

## Analysis Helper Packages

Zarr makes the output self-contained, but raw Zarr readers still leave too much
repetitive work for ordinary users. The midpoint guide should not require every
user to remember the tensor axis order, manually join group names, and rebuild
length/position metadata for each plot. The Python helper lives in
`py-cfdnalab`. The R helper lives in `r-cfdnalab`.

Proposed layout:

```text
py-cfdnalab/
  src/cfdnalab/
    midpoints.py
    ends.py
r-cfdnalab/
  R/
    midpoints.R
    end_motifs.R
    zarr_helpers.R
```

These helpers are not part of the Rust command implementation. They are thin
reader utilities for the public Zarr schema, and the downstream workflow should
test them against Zarr stores generated by cfDNAlab itself.

### Python Helper Contract

The Python package should expose one midpoint loader:

```python
import cfdnalab as cfl

profiles = cfl.read_midpoints("sample.midpoint_profiles.zarr")
```

Returned object responsibilities:

- validate `cfdnalab_schema == "midpoint_profiles"` and supported schema version
- expose `array()` for a materialized NumPy array
- expose `group_names`, `eligible_intervals`, length-bin metadata, and
  position-bin metadata as simple arrays/dataframes
- `array_from_group(group)` returns a 2D NumPy array
  `counts[length_bin, position]`, accepting either integer `group_idx` or
  `group_name`
- `array_from_length_bin(length_bin)` returns a 2D NumPy array
  `counts[group, position]`, accepting either integer bin index or a
  `(start_bp, end_bp)` pair
- `data_frame_from_group(group)` returns a long pandas dataframe with one row per
  `(length_bin, position)` cell
- `data_frame_from_length_bin(length_bin)` returns a long pandas dataframe with
  one row per `(group, position)` cell
- `data_frame_for_profile(group_idx, length_bin)` returns one row per position
  for a single profile

The dataframe columns should be stable and boring:

```text
group_idx
group_name
eligible_intervals
length_bin
length_start_bp
length_end_bp
position
position_bin_start_bp
position_bin_end_bp
count
```

The Python helper should use `zarr` as the base reader and return NumPy arrays
for selected profiles. It can optionally expose an `xarray()` method later, but
the first helper should not require xarray just to get a dataframe.

The Python package should also expose one ends loader:

```python
import cfdnalab as cfl

ends = cfl.read_end_motifs("sample.end_motifs.zarr")
```

Returned object responsibilities:

- validate `cfdnalab_schema == "end_motif_counts"` and supported schema version
- expose motif labels with `motifs()` and `motif_metadata()`
- return a mode-specific object based on `row_mode`
- build motif-indexed dataframes from every mode
- return SciPy COO sparse matrices through explicit `sparse_*` methods
- return NumPy arrays only through explicit `dense_*` methods
- reconstruct a dense matrix from sparse COO only when a `dense_*` method is
  requested

Mode-specific object responsibilities:

- `GlobalEndMotifCounts` exposes `dense_counts_vec()` and `dense_data_frame()`
- `WindowedEndMotifCounts` exposes `windows()`, `sparse_coo_for_window()`,
  `dense_counts_for_window()`, and `dense_data_frame_for_window()`
- `GroupedEndMotifCounts` exposes `groups()`, `group_idx()`,
  `sparse_coo_for_group()`, `dense_counts_for_group()`, and
  `dense_data_frame_for_group()`

The helper API should avoid generic `row` method names when the selected output
mode has a clearer user-facing concept such as window or group. It should also
avoid unqualified `array()` for ends because sparse-to-dense conversion must be
visible in the method name.

The helper tests should also cover loader behavior that matters for future R
parity:

- loader validation requires both the public `.zarr` suffix and the expected
  schema attrs
- required-array checks handle nested paths such as `sparse/row`
- sparse arrays carry expected dimension names, especially `nnz` and
  `sparse_dimension`
- schema-version checks follow the documented compatibility policy
- dense helpers document that they may load or reconstruct dense data

### R Helper Package Contract

The R package should be installed before use:

```r
library(cfdnalab)
profiles <- read_midpoints("sample.midpoint_profiles.zarr")
```

Returned object responsibilities:

- validate schema name and supported schema version where the chosen R reader
  exposes attrs
- keep the store path and read arrays lazily where practical
- expose `counts_array()` for users who want the full 3D array
- expose `group_metadata()`, `length_bins()`, and `position_bins()`
- `profile_for_group(profiles, group)` returns a matrix
  `counts[length_bin, position]`, accepting group index or group name
- `profiles_for_length_bin(profiles, length_bin)` returns a matrix
  `counts[group, position]`, accepting integer bin index or a two-element
  start/end vector
- `data_frame_for_group(profiles, group)` returns one row per
  `(length_bin, position)` cell
- `data_frame_for_length_bin(profiles, length_bin)` returns one row per
  `(group, position)` cell

Use base `data.frame` output first. Avoid making `tidyverse` a dependency for
the package.

### Plotting Boundary

Do not put plotting into the first helper packages. Add the data-frame builders
first and let users plot with their normal stack. If the helper API proves
stable and users repeatedly ask for standard plots, add optional plotting
functions later:

- Python: functions accepting matplotlib axes
- R: functions returning base plots or ggplot objects only if the dependency
  question is settled

### Package Graduation Criteria

Only turn sourceable R helpers into a package after the API has survived real
usage. A package is worth the overhead when:

- the helpers are used in the public guide
- downstream CI tests the helper API directly
- the midpoint schema is stable enough that the helper does not churn weekly
- ends Zarr output needs the same style of helper API
- users need versioned installs instead of copying/sourcing scripts

Possible R package names should be decided later. Do not let package naming
block sourceable R helpers.

## Midpoint Zarr Fixture

The fixture should be small but structurally complete:

```text
counts[group, length_bin, position]
group[group] with JSON attrs `label_field = "group_name"` and `labels`
eligible_intervals[group]
length_start_bp[length_bin]
length_end_bp[length_bin]
position_bin_start_bp[position]
position_bin_end_bp[position]
```

The current fixture uses:

- three groups
- two eligible intervals per group
- one eligible interval with no overlapping midpoint count
- three length bins
- five position bins
- multiple fragment lengths
- repeated intervals for the same group, so grouped accumulation is exercised
- varied nonzero count values from `--bin-size 2`
- ordinary ASCII group names plus longer names stored in JSON label attributes
- the same zstd-compressed Zarr V3 writer path used by `cfdna midpoints`

The fixture should be small enough that every test can also materialize a
dataframe slice.

## Required Python Checks

Python `zarr` checks:

- Store opens as Zarr V3.
- All expected arrays are present.
- Shapes and data types match the schema.
- Sliced reads return the expected values.
- Compression and chunking do not block reads.
- The Python helper package can load the store, return NumPy arrays, and build
  group-indexed and length-bin-indexed dataframes.

Python `xarray` checks:

- `xr.open_zarr()` succeeds.
- `counts` has dimensions `("group", "length_bin", "position")`.
- Coordinate arrays are visible as coordinates or loadable data variables.
- A small subset can be converted to a dataframe.
- The dataframe contains count, group, length-bin, and position metadata.

The xarray check is essential. A raw Zarr group that opens with `zarr` but fails
with `xarray` is not good enough for the midpoint use case.

Python `dask.array` checks:

- `da.from_zarr(..., component="counts")` succeeds.
- The chunked count tensor has the expected shape and dtype.
- A small slice can be computed without materializing the whole store.

## Required R Checks

CRAN `zarr` checks:

- Store opens as Zarr V3.
- Numeric arrays can be read.
- Sliced reads work.
- Group names can be read from JSON label attributes.
- A small R `data.frame` can be built from a slice.
- The `cfdnalab` R package can load midpoint and end-motif stores generated by
  cfDNAlab and return expected data frames, arrays, and sparse matrices.

Bioconductor `Rarr` checks:

- Numeric arrays can be read.
- Dimension metadata does not prevent reading.
- The selected compression codec works.
- JSON label attributes can be read from array metadata.

Future package candidates:

- `ZarrArray`, if the package exposes a practical array-level read path for this
  hierarchy.
- `pizzarr`, if V3 string and zstd support become complete enough for this
  schema.

The R checks should avoid assuming that every package exposes the same object
model. The contract is successful loading and reconstruction, not identical R
APIs.

## Required Ends Checks

The ignored Rust fixture-generation test runs the real `cfdna ends` command and
creates three tiny stores:

```text
tiny_dense_global.end_motifs.zarr/
tiny_sparse_windowed.end_motifs.zarr/
tiny_sparse_grouped.end_motifs.zarr/
```

Together the fixtures should contain:

- global, windowed, and grouped row modes
- dense and sparse COO storage modes
- repeated group accumulation
- at least one empty group row
- motif labels stored as fixed-width `motif_ascii[motif, motif_byte]`
- row metadata for ordinary windows and grouped BED output

The sparse fixtures should contain COO matrices with:

- sorted `(row, motif)` coordinates
- at least one missing row/motif pair that reconstructs as zero
- a grouped row with no counts, so empty rows remain visible through metadata

Python ends checks:

- `zarr` opens both dense and sparse stores.
- Dense `counts[row, motif]` has expected shape, dtype, and values.
- Sparse COO arrays reconstruct a `scipy.sparse.coo_matrix` with the expected
  dense values.
- Motif labels decoded from `motif_ascii` and row metadata can be joined into a
  dataframe.
- The Python helper returns the expected mode-specific class and exposes
  window/group/global methods only where they make sense.

R ends checks:

- `Rarr` reads dense counts, sparse COO arrays, motif labels from `motif_ascii`,
  and row metadata.
- Sparse COO arrays reconstruct the expected count matrix after converting
  zero-based coordinates to one-based R indices.
- Motif labels decode without depending on Zarr string arrays or JSON attrs.
- The `cfdnalab` R package returns the expected dense global counts, sparse
  windowed matrix, sparse grouped matrix, and mode-specific metadata helpers.

Do not require `xarray` for sparse ends unless it gives a natural user workflow.
Raw Zarr plus `scipy.sparse` / `Matrix` is the more direct downstream contract
for COO payloads.

## Zarr V3 Safe Subset To Test First

Start with a deliberately small Zarr V3 subset:

- regular chunk grid only
- no sharding
- no storage transformers
- no extension data types
- no structured arrays
- no datetime or timedelta arrays
- no complex arrays
- numeric arrays limited to `float32`, `float64`, `int32`, `uint32`, and
  `uint64`
- avoid public `uint64` unless values can exceed `int32`
- V3 `dimension_names` metadata on every array
- coordinate arrays stored as arrays, not only as attrs

For codecs:

- Implemented fixture: zstd via cfDNAlab's `zarrs` writer.
- Add an uncompressed baseline only if a reader fails and the failure needs to
  be separated from codec support.
- Preferred production candidate: zstd.

Zstd is the likely production choice if Python, CRAN `zarr`, and Bioconductor
readers all pass. It is fast and compresses numeric scientific arrays well. The
compatibility test may still need a boring baseline if a reader fails, because
that separates "the schema is wrong" from "this reader cannot handle this
codec".

## Label Encoding

Public labels are stored as JSON attributes on numeric coordinate arrays:

```json
{
  "label_field": "group_name",
  "labels": ["A", "B"]
}
```

This applies to midpoint group names, ends chromosome names, ends group names,
global row labels, and sparse dimension names. Downstream tests must verify that
Python and R readers can access those attrs without relying on Zarr string
arrays.

High-cardinality fixed-width labels should be stored as byte arrays instead of
JSON attrs. End-motif labels use:

```text
motif_index            int32[motif]
motif_byte             int32[motif_byte]
motif_ascii            uint8[motif, motif_byte]
```

The downstream tests must decode `motif_ascii` in Python and R and verify that
the decoded labels match the count-matrix motif axis.

## R Integer Strategy

The R tests must check integer conversion deliberately instead of accepting the
reader package default. Public Zarr schemas use unsigned integer dtypes where
that is the correct on-disk domain type:

- genomic coordinates such as `row_start_bp` and `row_end_bp` may be `uint64`
- sparse coordinates such as `sparse/row`, `sparse/motif`, and `sparse/shape`
  are `int32`
- counts of windows or intervals should be `int32` when they are intended to
  become ordinary R integers

The R helper should read exact `uint64` values as `bit64::integer64` when they
are retained as metadata. When constructing `Matrix::sparseMatrix`, sparse
coordinates should be range-checked and shifted from zero-based Zarr coordinates
to one-based R indices.

The public schema spec should list every array dtype and the intended R cast so
the R package does not have to infer policy from examples.

## Chunking Checks

The compatibility fixture should not settle performance chunking, but it should
exercise chunked reads. Use chunks that split at least one dimension:

```text
counts chunks: [1, 1, 2]
```

The production midpoint writer can choose larger chunks later, probably biased
toward reading a small number of groups across all length bins and positions.

## Pass Criteria

The downstream test suite passes when:

- Python `zarr` can read all arrays.
- Python `xarray` sees named dimensions on `counts`.
- R can read counts and coordinate arrays without Python.
- R can read group names from JSON label attributes.
- R and Python can decode fixed-width end-motif labels from byte arrays.
- zstd-compressed fixtures pass in every supported reader.
- A small dataframe can be built in both Python and R.

## Non-Goals

- Do not test every R package that might read Zarr.
- Do not make this part of the normal local `cargo check` workflow.
- Do not benchmark compression or chunking here.
- Do not test large production-sized outputs in this workflow.
- Do not require local developers to install R, Bioconductor, or Python
  scientific stacks just to work on unrelated Rust code.
- Do not show every tested package in the user guide. Users should see one
  concise Python example and one concise R example.

# Downstream Compatibility Testing

This note sketches CI-only tests for public output formats. These tests should
verify that files written by `cfDNAlab` can be loaded by the downstream tools
users are likely to use, without making those tools part of the normal local
Rust development environment.

## Why This Belongs In Separate CI

The normal development loop should stay Rust-focused:

```text
cargo check
cargo check --tests
```

R packages, Python scientific stacks, system libraries, and Bioconductor release
timing are a different dependency surface. They are still important because the
public output contract is only useful if real downstream readers can load it.
The right compromise is a separate GitHub Actions workflow that runs on demand,
on pushes to `main`, and on PRs that touch public output code or downstream
fixtures.

## Implemented Layout

```text
downstream_tests/
  README.md
  requirements-python.txt
  python/
    conftest.py
    test_midpoint_zarr_python.py
    test_midpoint_zarr_xarray.py
    test_midpoint_zarr_dask.py
  R/
    common.R
    test_midpoint_zarr_cran_zarr.R
    test_midpoint_zarr_rarr.R
tests/
  generate_downstream_midpoint_fixture.rs
```

`generate_downstream_midpoint_fixture.rs` creates a small BAM plus interval
file, runs the real `cfdna midpoints` command, and leaves the resulting Zarr
store under `downstream_tests/tmp`. This is required: the downstream suite
exists to catch cfDNAlab output-schema drift, so the file consumed by Python and
R must be written by cfDNAlab itself.

The Python and R tests should fail when the cfDNAlab-generated fixture is
missing. They should not create a hand-authored Zarr store as a fallback.

When ends moves to Zarr, add separate dense and sparse fixtures rather than
folding them into the midpoint fixture. Ends compatibility has a different
success condition: reconstructing motif matrices and sparse matrices with row
and motif metadata intact.

## GitHub Actions Workflow

Use a separate workflow:

```text
.github/workflows/downstream-zarr.yml
```

Suggested triggers:

```yaml
on:
  workflow_dispatch:
  push:
    branches:
      - main
  pull_request:
    paths:
      - "src/commands/midpoints/**"
      - "src/shared/**zarr**"
      - "tests/generate_downstream_midpoint_fixture.rs"
      - "tests/fixtures/**"
      - "downstream_tests/**"
      - ".github/workflows/downstream-zarr.yml"
```

Run the workflow on pushes to `main` and relevant pull requests. Do not schedule
it with cron. The suite should be small enough that running it on code changes is
the right signal, and scheduled package-drift failures are likely to be noise
unless the output code is being changed.

Keep the workflow optional at first. Once the midpoint Zarr writer and supported
downstream package set are stable, it can become required for PRs that touch the
Zarr writer or output schema.

The first workflow jobs are:

- Fixture generation: Rust ignored integration test that runs `cfdna midpoints`
- Python: `zarr`, `xarray`, and `dask.array`
- R: CRAN `zarr` and Bioconductor `Rarr`

The R job installs `libblosc-dev` and `libzstd-dev` before installing R
packages. Those libraries make compression support explicit instead of relying
on the GitHub runner image.

Do not let package-sprawl leak into the user docs. CI can test several readers,
but the guide should show one recommended Python path and one recommended R
path.

## Sourceable Analysis Helpers

Zarr makes the output self-contained, but raw Zarr readers still leave too much
repetitive work for ordinary users. The midpoint guide should not require every
user to remember the tensor axis order, manually join group names, and rebuild
length/position metadata for each plot. Before making real Python or R packages,
add small sourceable helper scripts that define the analysis-facing API we would
eventually package.

Proposed layout:

```text
downstream_tests/
  helpers/
    python/
      cfdnalab_midpoints.py
    R/
      cfdnalab_midpoints.R
```

These scripts are not part of the Rust command implementation. They are thin
reader utilities for the public Zarr schema, and the downstream workflow should
test them against a Zarr store generated by cfDNAlab itself.

### Python Helper Contract

The Python helper should expose one loader:

```python
from cfdnalab_midpoints import read_midpoint_profiles

profiles = read_midpoint_profiles("sample.midpoint_profiles.zarr")
```

Returned object responsibilities:

- validate `cfdnalab_schema == "midpoint_profiles"` and supported schema version
- expose `counts` as the raw NumPy-like array reader
- expose `counts_array()` for a materialized NumPy array
- expose `group_names`, `eligible_intervals`, length-bin metadata, and
  position-bin metadata as simple arrays/dataframes
- `profile_for_group(group)` returns a 2D NumPy array
  `counts[length_bin, position]`, accepting either integer `group_idx` or
  `group_name`
- `profiles_for_length_bin(length_bin)` returns a 2D NumPy array
  `counts[group, position]`, accepting either integer bin index or a
  `(start_bp, end_bp)` pair
- `dataframe_for_group(group)` returns a long pandas dataframe with one row per
  `(length_bin, position)` cell
- `dataframe_for_length_bin(length_bin)` returns a long pandas dataframe with
  one row per `(group, position)` cell

The dataframe columns should be stable and boring:

```text
group_idx
group_name
eligible_intervals
length_bin
length_start_bp
length_end_bp
position
position_bin_start_bp
position_bin_end_bp
count
```

The Python helper should use `zarr` as the base reader and return NumPy arrays
for selected profiles. It can optionally expose an `xarray()` method later, but
the first helper should not require xarray just to get a dataframe.

### R Helper Contract

The R helper should be sourceable:

```r
source("cfdnalab_midpoints.R")
profiles <- read_midpoint_profiles("sample.midpoint_profiles.zarr")
```

Returned object responsibilities:

- validate schema name and supported schema version where the chosen R reader
  exposes attrs
- keep the store path and read arrays lazily where practical
- expose `counts_array()` for users who want the full 3D array
- expose `group_metadata()`, `length_bins()`, and `position_bins()`
- `profile_for_group(profiles, group)` returns a matrix
  `counts[length_bin, position]`, accepting group index or group name
- `profiles_for_length_bin(profiles, length_bin)` returns a matrix
  `counts[group, position]`, accepting integer bin index or a two-element
  start/end vector
- `data_frame_for_group(profiles, group)` returns one row per
  `(length_bin, position)` cell
- `data_frame_for_length_bin(profiles, length_bin)` returns one row per
  `(group, position)` cell

Use base `data.frame` output first. Avoid making `tidyverse` a dependency for a
sourceable script.

### Plotting Boundary

Do not put plotting into the first helper scripts. Add the data-frame builders
first and let users plot with their normal stack. If the helper API proves stable
and users repeatedly ask for standard plots, add optional plotting functions
later:

- Python: functions accepting matplotlib axes
- R: functions returning base plots or ggplot objects only if the dependency
  question is settled

### Package Graduation Criteria

Only turn these scripts into packages after the sourceable API has survived real
usage. A package is worth the overhead when:

- the helper scripts are used in the public guide
- downstream CI tests the helper API directly
- the midpoint schema is stable enough that the helper does not churn weekly
- ends Zarr output needs the same style of helper API
- users need versioned installs instead of copying/sourcing scripts

Possible package names should be decided later. Do not let package naming block
the sourceable-script experiment.

## Midpoint Zarr Fixture

The fixture should be small but structurally complete:

```text
counts[group, length_bin, position]
group_name[group]
group_name_utf8[group, max_name_bytes]
group_name_nbytes[group]
eligible_intervals[group]
length_start_bp[length_bin]
length_end_bp[length_bin]
position_bin_start_bp[position]
position_bin_end_bp[position]
```

The current fixture uses:

- three groups
- two eligible intervals per group
- one eligible interval with no overlapping midpoint count
- three length bins
- five position bins
- multiple fragment lengths
- repeated intervals for the same group, so grouped accumulation is exercised
- varied nonzero count values from `--bin-size 2`
- ordinary ASCII group names plus longer names that exercise the padded byte
  fallback
- the same zstd-compressed Zarr V3 writer path used by `cfdna midpoints`

The fixture should be small enough that every test can also materialize a
dataframe slice.

## Required Python Checks

Python `zarr` checks:

- Store opens as Zarr V3.
- All expected arrays are present.
- Shapes and data types match the schema.
- Sliced reads return the expected values.
- Compression and chunking do not block reads.
- The sourceable Python helper can load the store, return raw NumPy arrays, and
  build group-indexed and length-bin-indexed dataframes.

Python `xarray` checks:

- `xr.open_zarr()` succeeds.
- `counts` has dimensions `("group", "length_bin", "position")`.
- Coordinate arrays are visible as coordinates or loadable data variables.
- A small subset can be converted to a dataframe.
- The dataframe contains count, group, length-bin, and position metadata.

The xarray check is essential. A raw Zarr group that opens with `zarr` but fails
with `xarray` is not good enough for the midpoint use case.

Python `dask.array` checks:

- `da.from_zarr(..., component="counts")` succeeds.
- The chunked count tensor has the expected shape and dtype.
- A small slice can be computed without materializing the whole store.

## Required R Checks

CRAN `zarr` checks:

- Store opens as Zarr V3.
- Numeric arrays can be read.
- Sliced reads work.
- Group names can be read or reconstructed from the fallback encoding.
- A small R `data.frame` can be built from a slice.
- The sourceable R helper can load the store and build group-indexed and
  length-bin-indexed data frames.

Bioconductor `Rarr` checks:

- Numeric arrays can be read.
- Dimension metadata does not prevent reading.
- The selected compression codec works.
- String arrays work, or the fallback byte encoding can be decoded.

Future package candidates:

- `ZarrArray`, if the package exposes a practical array-level read path for this
  hierarchy.
- `pizzarr`, if V3 string and zstd support become complete enough for this
  schema.

The R checks should avoid assuming that every package exposes the same object
model. The contract is successful loading and reconstruction, not identical R
APIs.

## Future Ends Checks

Add an ignored Rust fixture-generation test only after the ends Zarr schema is
implemented. It should run the real `cfdna ends` command and create two tiny
stores:

```text
tiny.end_motifs.dense.zarr/
tiny.end_motifs.sparse.zarr/
```

Both fixtures should contain:

- two rows
- three motifs
- non-integer counts
- one zero dense count
- motif string array plus byte fallback
- enough row metadata to cover one ordinary window mode

The sparse fixture should contain a COO matrix with:

- sorted `(row, motif)` coordinates
- at least one nonzero value in each row
- one missing row/motif pair that reconstructs as zero

Python ends checks:

- `zarr` opens both dense and sparse stores.
- Dense `counts[row, motif]` has expected shape, dtype, and values.
- Sparse COO arrays reconstruct a `scipy.sparse.coo_matrix` with the expected
  dense values.
- Motif labels and row metadata can be joined into a dataframe.

R ends checks:

- `Rarr` reads dense counts, sparse COO arrays, motif labels, and row metadata.
- Sparse COO arrays reconstruct a `Matrix::sparseMatrix` after converting
  zero-based coordinates to one-based R indices.
- The byte fallback decodes motif labels if native Zarr strings fail.

Do not require `xarray` for sparse ends unless it gives a natural user workflow.
Raw Zarr plus `scipy.sparse` / `Matrix` is the more direct downstream contract
for COO payloads.

## Zarr V3 Safe Subset To Test First

Start with a deliberately small Zarr V3 subset:

- regular chunk grid only
- no sharding
- no storage transformers
- no extension data types
- no structured arrays
- no datetime or timedelta arrays
- no complex arrays
- numeric arrays limited to `float32`, `int32`, and `uint32`
- avoid public `uint64` unless values can exceed `int32`
- V3 `dimension_names` metadata on every array
- coordinate arrays stored as arrays, not only as attrs

For codecs:

- Implemented fixture: zstd via cfDNAlab's `zarrs` writer.
- Add an uncompressed baseline only if a reader fails and the failure needs to
  be separated from codec support.
- Preferred production candidate: zstd.

Zstd is the likely production choice if Python, CRAN `zarr`, and Bioconductor
readers all pass. It is fast and compresses numeric scientific arrays well. The
compatibility test may still need a boring baseline if a reader fails, because
that separates "the schema is wrong" from "this reader cannot handle this
codec".

## Group Name Encoding Matrix

Test two options before choosing the public schema.

Native string array:

```text
group_name              string[group]
```

Fallback byte encoding:

```text
group_name_utf8         uint8[group, max_name_bytes]
group_name_nbytes       uint32[group]
```

Use native strings only if they pass in Python `zarr`, Python `xarray`, CRAN
`zarr`, and Bioconductor `Rarr`. If not, use the byte encoding. It is less
pleasant, but it is explicit and easy to reconstruct everywhere.

The first docs should show the primary `group_name` path if it works in the
chosen package. Keep fallback decoding in CI so the schema remains recoverable
if a reader loses native string support.

## Chunking Checks

The compatibility fixture should not settle performance chunking, but it should
exercise chunked reads. Use chunks that split at least one dimension:

```text
counts chunks: [1, 1, 2]
```

The production midpoint writer can choose larger chunks later, probably biased
toward reading a small number of groups across all length bins and positions.

## Pass Criteria

The downstream test suite passes when:

- Python `zarr` can read all arrays.
- Python `xarray` sees named dimensions on `counts`.
- R can read counts and coordinate arrays without Python.
- R can read or reconstruct group names.
- zstd-compressed fixtures pass in every supported reader.
- A small dataframe can be built in both Python and R.

## Non-Goals

- Do not test every R package that might read Zarr.
- Do not make this part of the normal local `cargo check` workflow.
- Do not benchmark compression or chunking here.
- Do not test large production-sized outputs in this workflow.
- Do not require local developers to install R, Bioconductor, or Python
  scientific stacks just to work on unrelated Rust code.
- Do not show every tested package in the user guide. Users should see one
  concise Python example and one concise R example.
